# SocialFlow 功能流程详解

> SocialFlow -- AI 社交媒体内容运营平台
>
> 技术栈: Spring Boot 3.3 + JDK 21 + Vue 3 + TypeScript + Element Plus
>
> AI 能力: DeepSeek (reasoner/chat) 主力大模型, Qwen 文本嵌入
>
> 数据层: PostgreSQL pgvector (向量检索), MySQL (关系数据), Redis (缓存), 腾讯云 COS (对象存储)

## 1. AI 内容生成

### 1.1 功能概述

AI 内容生成是 SocialFlow 平台的核心功能, 支持用户输入主题、关键词、平台等参数后, 由 AI 自动生成适合目标社交媒体平台的文案内容。系统提供四种生成模式: 普通生成、流式生成 (SSE)、批量生成、RAG 增强生成, 以及多 Agent 协作生成。

### 1.2 REST 端点

| 方法 | 路径 | 说明 | 返回类型 |
|------|------|------|----------|
| POST | `/content/generate` | 普通生成, 同步返回完整结果 | `Result<ContentVO>` |
| POST | `/content/generate-stream` | 流式生成, SSE 逐 token 推送 | `Flux<String>` (SSE) |
| POST | `/content/generate-batch` | 批量生成, 同时为多个平台生成内容 | `Result<List<ContentVO>>` |
| POST | `/content/generate-with-rag` | RAG 增强生成, 结合知识库上下文 | `Result<ContentVO>` |
| POST | `/content/generate-multi-agent` | 多 Agent 协作生成 | `Flux<String>` (SSE) |

### 1.3 请求参数

```json
{
  "topic": "春季护肤技巧",
  "keywords": ["防晒", "保湿", "美白"],
  "platform": "XIAOHONGSHU",
  "tone": "轻松活泼",
  "wordCount": 500,
  "templateId": 1001,
  "knowledgeBaseId": 2001,
  "productInfo": "XX品牌防晒霜SPF50+"
}
```

### 1.4 五步生成管线

```text
┌─────────────────────────────────────────────────────────────────────┐
│                     ContentServiceImpl.java                         │
│                                                                     │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌────────┐   ┌────┐ │
│  │ 1.输入   │──>│ 2.RAG    │──>│ 3.提示词  │──>│ 4.LLM  │──>│5.输│ │
│  │   安全   │   │   检索   │   │   渲染   │   │   调用  │   │出检│ │
│  │   检查   │   │          │   │          │   │        │   │查+ │ │
│  │          │   │          │   │          │   │        │   │保存│ │
│  └──────────┘   └──────────┘   └──────────┘   └────────┘   └────┘ │
│       │              │              │              │           │    │
│  GuardrailSvc   RagPipelineSvc  PromptSvc     LlmRouter    Guard  │
│  .checkInput()  .retrieve()     .render()     .chat()      +Save  │
└─────────────────────────────────────────────────────────────────────┘
```

**步骤详解:**

**第一步: 输入安全检查 (Guardrail Check)**
- 调用 `GuardrailService.checkInput(userInput)`
- 敏感词词典过滤: 遍历预加载的敏感词集合, 检测用户输入是否包含违规词汇
- 输入长度校验: 确保输入不超过系统设定的最大长度限制
- 若检查不通过, 直接返回错误信息, 不进入后续流程

**第二步: RAG 检索 (可选)**
- 当请求中包含 `knowledgeBaseId` 时触发
- 调用 `RagPipelineServiceImpl.retrieve(query, knowledgeBaseId)`
- 返回相关知识库文档片段作为生成上下文 (详见第 3 节)

**第三步: 提示词渲染**
- 调用 `PromptServiceImpl.render(templateId, platform, variables, ragContext)`
- 将模板中的占位符替换为实际值: `{{topic}}`, `{{keywords}}`, `{{wordCount}}` 等
- 注入 RAG 上下文到提示词中
- 如未选择模板, 使用系统内置的 fallback 模板

**第四步: LLM 调用**
- 通过 `LlmRouter` 路由到对应的 LLM Provider
- 默认使用 DeepSeek deepseek-chat 模型
- 普通模式: 同步调用, 等待完整响应
- 流式模式: 调用 streaming API, 返回 `Flux<String>`

**第五步: 输出检查 + 保存**
- 调用 `GuardrailService.checkOutput(generatedContent)` 进行输出安全检查
- LLM 辅助的输出质量检查
- 保存内容到 `content` 表
- 自动提取图片提示词 (用于后续 AI 配图)
- 调用嵌入服务生成向量, 存入 `sf_content_vectors` 集合

### 1.5 流式生成 (SSE) 数据流

```text
前端                                后端                              LLM
 │                                   │                                 │
 │  POST /content/generate-stream    │                                 │
 │ ─────────────────────────────────>│                                 │
 │                                   │  步骤1-3: 安全检查+RAG+渲染     │
 │                                   │────────────────────────────────>│
 │                                   │                                 │
 │                                   │  Flux<String> streaming tokens  │
 │                                   │<────────────────────────────────│
 │   SSE: {"token":"春"}             │                                 │
 │<──────────────────────────────────│                                 │
 │   SSE: {"token":"季"}             │                                 │
 │<──────────────────────────────────│                                 │
 │   SSE: {"token":"护"}             │                                 │
 │<──────────────────────────────────│                                 │
 │   ...逐token推送...               │                                 │
 │   SSE: {"done":true}              │                                 │
 │<──────────────────────────────────│                                 │
 │                                   │  步骤5: 输出检查 + 保存          │
 │                                   │                                 │
```

**前端处理 (useSse.ts):**
- 使用 fetch API 建立 SSE 连接 (非 EventSource, 因需携带 Bearer Token)
- 请求头: `Authorization: Bearer <token>`, `Accept: text/event-stream`
- 逐条解析 SSE 事件, 累积 token 拼接为完整文本
- 实时渲染到 Workspace 页面的编辑区域
- 收到 `{"done":true}` 后关闭连接, 触发后续保存逻辑

### 1.6 批量生成模式

```text
┌─────────────────────────────────────────────────────────────┐
│                    批量生成流程                               │
│                                                             │
│  请求: platforms = [XIAOHONGSHU, DOUYIN, WECHAT_MP]         │
│                                                             │
│  ┌─────────────────────┐                                    │
│  │ CompletableFuture   │                                    │
│  │ .supplyAsync()      │                                    │
│  │                     │                                    │
│  │  ┌───────────┐      │                                    │
│  │  │ 小红书生成  │─────>│─── 五步管线 ─── ContentVO        │
│  │  └───────────┘      │                                    │
│  │  ┌───────────┐      │                                    │
│  │  │ 抖音生成   │─────>│─── 五步管线 ─── ContentVO        │
│  │  └───────────┘      │                                    │
│  │  ┌───────────┐      │                                    │
│  │  │ 公众号生成  │─────>│─── 五步管线 ─── ContentVO        │
│  │  └───────────┘      │                                    │
│  └─────────────────────┘                                    │
│                                                             │
│  CompletableFuture.allOf() 等待全部完成                      │
│  返回: List<ContentVO>                                      │
└─────────────────────────────────────────────────────────────┘
```

- 使用 `CompletableFuture` 并行执行多个平台的生成任务
- 每个平台独立执行完整的五步管线
- 所有任务完成后统一返回结果列表
- 单个平台失败不影响其他平台的生成

### 1.7 生成后处理

- **图片提示词提取**: 调用 DeepSeek 从生成内容中提取适合配图的中文描述
- **数据库保存**: 内容存入 `content` 表, 包含 title, body, platform, status 等字段
- **版本记录**: 同时写入 `content_version` 表, 记录每次生成的版本
- **向量嵌入**: 调用 Qwen text-embedding-v3 生成 1024 维向量, 存入 PostgreSQL pgvector

| ContentDTO.java | `dto/ContentDTO.java` | 内容生成请求参数 |
| ContentVO.java | `vo/ContentVO.java` | 内容返回视图对象 |

## 2. 多 Agent 协作

### 2.1 功能概述

多 Agent 协作模式是 SocialFlow 的高级内容生成功能。系统模拟一个专业内容团队, 由四个专业 Agent 分工协作, 通过多轮迭代产出高质量内容。每个 Agent 扮演不同角色, 形成"策划-写作-审核-优化"的完整工作流。

### 2.2 四阶段管线

```text
┌──────────────────────────────────────────────────────────────────┐
│              MultiAgentServiceImpl.java                          │
│                                                                  │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│   │ Planner  │───>│ Writer   │───>│ Reviewer │───>│Optimizer │  │
│   │ (策划)   │    │ (写作)   │    │ (审核)   │    │ (优化)   │  │
│   └──────────┘    └──────────┘    └──────────┘    └──────────┘  │
│        │               │               │    │          │         │
│   制定策略方案      根据策略撰写       评审质量    │    根据反馈优化  │
│   分析目标受众      初稿内容          给出评分    │    迭代改进      │
│   明确创作方向                       提出修改建议 │                 │
│                                          │       │                 │
│                                     通过? ─┤     │                 │
│                                     是──────────>│ 输出最终内容     │
│                                     否──────────>│ 驳回, 重新优化   │
│                                          ▲       │                 │
│                                          │       │                 │
│                                          └───────┘                 │
│                                        (最多 N 轮迭代)             │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 各 Agent 职责

**Planner (策划 Agent)**
- 分析用户输入的主题、平台、受众等信息
- 制定内容策略方案: 写作角度、结构大纲、关键卖点
- 输出策略文档供 Writer 参考
- SSE 事件: `{"stage":"PLANNER","message":"策略制定中..."}`

**Writer (写作 Agent)**
- 根据 Planner 的策略方案进行内容创作
- 产出符合目标平台风格的初稿
- 写作过程逐 token 推送到前端
- SSE 事件: `{"token":"..."}`

**Reviewer (审核 Agent)**
- 从多个维度评审 Writer 的初稿
- 评审维度: 内容相关性、平台适配性、文案质量、创意性
- 给出评分和具体修改建议
- 决定是否通过: 通过则流转到最终输出, 不通过则转交 Optimizer
- SSE 事件: `{"stage":"REVIEWER","message":"审核反馈..."}`

**Optimizer (优化 Agent)**
- 接收 Reviewer 的反馈意见
- 对内容进行针对性优化和修改
- 优化后的内容重新提交给 Reviewer 审核
- 优化过程逐 token 推送到前端
- SSE 事件: `{"token":"..."}`

### 2.4 迭代控制

- 最大迭代轮次通过配置参数 `maxRounds` 控制
- Reviewer 驳回后, Optimizer 进行优化, 再次提交 Reviewer
- 达到最大轮次后强制输出当前最优版本
- 每轮迭代的 Reviewer 反馈都会传递给 Optimizer 作为优化依据

### 2.5 SSE 事件格式

```text
前端 SSE 接收的事件流:

event: stage
data: {"stage":"PLANNER","message":"正在制定内容策略..."}

event: stage
data: {"stage":"PLANNER","message":"策略方案: 1.以测评角度切入 2.突出成分对比..."}

event: token
data: {"token":"春"}

event: token
data: {"token":"天"}

event: token
data: {"token":"到"}
...Writer 逐 token 输出初稿...

event: stage
data: {"stage":"REVIEWER","message":"评审意见: 开头吸引力不足, 建议增加痛点场景..."}

event: token
data: {"token":"你"}
...Optimizer 逐 token 输出优化版...

event: stage
data: {"stage":"REVIEWER","message":"评审通过, 内容质量达标"}

event: done
data: {"done":true,"contentId":123456}
```

**前端处理逻辑:**
- 仅 Writer 和 Optimizer 的 token 事件渲染到文本编辑区
- Planner 和 Reviewer 的 stage 事件显示在状态面板中
- 前端展示当前所处阶段的进度指示器
- 每次 Optimizer 输出替换上一轮的内容

### 2.6 数据流

```text
前端                        后端                              LLM
 │                           │                                 │
 │ POST /generate-multi-agent│                                 │
 │ ─────────────────────────>│                                 │
 │                           │                                 │
 │                           │  Planner 策划                    │
 │                           │────────────────────────────────>│
 │  SSE: stage PLANNER       │<────────────────────────────────│
 │<──────────────────────────│                                 │
 │                           │  Writer 写作                     │
 │                           │────────────────────────────────>│
 │  SSE: token (逐字)        │<────────────────────────────────│
 │<──────────────────────────│                                 │
 │                           │  Reviewer 审核                   │
 │                           │────────────────────────────────>│
 │  SSE: stage REVIEWER      │<────────────────────────────────│
 │<──────────────────────────│                                 │
 │                           │  [若驳回] Optimizer 优化          │
 │                           │────────────────────────────────>│
 │  SSE: token (逐字)        │<────────────────────────────────│
 │<──────────────────────────│                                 │
 │                           │  [循环直到通过或达到最大轮次]      │
 │                           │                                 │
 │  SSE: done                │  保存最终内容到数据库              │
 │<──────────────────────────│                                 │
```

## 3. RAG 知识库增强生成

### 3.1 功能概述

RAG (Retrieval-Augmented Generation) 知识库增强生成允许用户创建专属知识库, 上传产品资料、品牌规范、行业报告等文档, 在内容生成时自动检索相关知识片段注入提示词, 使 AI 生成的内容更加准确、专业, 避免幻觉。

### 3.2 REST 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/kb` | 创建知识库 |
| GET | `/kb` | 获取知识库列表 |
| GET | `/kb/{id}` | 获取知识库详情 |
| PUT | `/kb/{id}` | 更新知识库信息 |
| DELETE | `/kb/{id}` | 删除知识库 |
| POST | `/kb/{id}/upload` | 上传文档到知识库 |
| POST | `/kb/{id}/search` | 搜索知识库内容 |
| GET | `/kb/{id}/docs` | 获取知识库文档列表 |
| DELETE | `/kb/{id}/docs/{docId}` | 删除知识库文档 |

### 3.4 八步检索管线

```text
┌───────────────────────────────────────────────────────────────────────┐
│                 RagPipelineServiceImpl.java                           │
│                         八步检索流程                                   │
│                                                                       │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐                     │
│  │1.HyDE  │─>│2.Embed │─>│3.向量  │─>│4.BM25  │                     │
│  │查询改写 │  │ding    │  │  检索  │  │ 全文   │                     │
│  │        │  │ 向量化  │  │pgvector│  │ 检索   │                     │
│  └────────┘  └────────┘  └────────┘  └────────┘                     │
│       │           │           │           │                          │
│  DeepSeek    Qwen emb     余弦相似度   MySQL FULLTEXT               │
│  改写查询    1024维向量    Top-K 结果   Top-K 结果                    │
│                                                                       │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐                     │
│  │5.RRF   │─>│6.Rerank│─>│7.上下文│─>│8.注入  │                     │
│  │ 融合   │  │ 重排序  │  │  组装  │  │ 提示词 │                     │
│  └────────┘  └────────┘  └────────┘  └────────┘                     │
│       │           │           │           │                          │
│  倒数排名融合  可选步骤     拼接文档片段  合并到最终                   │
│  合并两路结果  LLM重排     添加来源标注  prompt中                     │
└───────────────────────────────────────────────────────────────────────┘
```

**各步骤详解:**

**步骤 1: HyDE 查询改写 (Hypothetical Document Embeddings)**
- 将用户原始查询发送给 DeepSeek, 生成一段假设性回答文本
- 目的: 改写后的查询在语义空间中更接近目标文档
- 示例: 用户查询"防晒霜用法" -> 改写为"SPF50+防晒霜建议在出门前30分钟均匀涂抹..."

**步骤 2: Embedding 向量化**
- 使用 Qwen text-embedding-v3 模型将改写后的查询转为 1024 维向量
- 专用 API Key 配置, 与 LLM 调用分开计费
- 调用 DashScope 兼容 API

**步骤 3: pgvector 向量检索**
- 在 PostgreSQL pgvector 的 `sf_kb_chunks` 集合中执行余弦相似度搜索
- 返回 Top-K 条最相似的文档分块 (默认 K=10)
- 过滤条件: 限定在指定知识库 ID 范围内

**步骤 4: BM25 全文检索**
- 在 MySQL `kb_chunk` 表上执行 FULLTEXT 全文检索
- 使用原始查询关键词进行匹配
- 返回 Top-K 条结果作为补充
- 目的: 与向量检索互补, 捕获精确关键词匹配

**步骤 5: RRF 融合 (Reciprocal Rank Fusion)**
- 将向量检索和 BM25 检索的结果进行融合排序
- RRF 公式: `score = sum(1 / (k + rank_i))`, k=60
- 合并去重后按融合分数降序排列
- 取 Top-N 条结果进入下一步

**步骤 6: Rerank 重排序 (可选)**
- 可选步骤, 通过配置开关控制是否启用
- 使用 LLM 对候选文档片段进行精细化重排序
- 综合考虑查询与文档的语义相关性
- 进一步提升检索精度

**步骤 7: 上下文组装**
- 将排序后的文档片段拼接为结构化上下文
- 添加来源标注: 文档名称、分块位置等元信息
- 控制总上下文长度, 避免超出 LLM 上下文窗口

**步骤 8: 注入提示词**
- 将组装好的 RAG 上下文注入到提示词模板的 `{{ragContext}}` 位置
- 提示词中包含指令: "请参考以下知识库资料进行回答, 如资料与问题无关请忽略"
- 最终提示词传递给 LLM 进行内容生成

## 4. AI 智能配图

### 4.1 功能概述

AI 智能配图功能在内容生成后, 自动调用 DeepSeek 从文案中提取适合配图的中文描述作为图片提示词, 然后调用阿里云 DashScope 的 wanx-v1 文生图模型异步生成图片。生成完成后自动下载到 COS 存储, 并关联到对应内容。

### 4.2 REST 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/image/extract-prompt` | 从内容提取图片提示词 |
| POST | `/image/generate` | 提交图片生成任务 |
| GET | `/image/generate/status/{taskId}` | 查询生成任务状态 |
| POST | `/image/download` | 下载生成的图片到 COS |

### 4.3 完整数据流

```text

│                                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐          │
│  │1.提取    │───>│2.用户    │───>│3.提交    │───>│4.轮询    │          │

│  │  提示词  │    │  提示词  │    │  任务    │    │         │          │

│  图片文件        素材元数据       内容-素材关联                          │
└─────────────────────────────────────────────────────────────────────────┘
```

**步骤详解:**

**第 1 步: 提取图片提示词**
- 内容生成完成后, 调用 `POST /image/extract-prompt`
- 后端将内容文本发送给 DeepSeek, 要求提取适合配图的中文描述
- 提示词示例: "请从以下文案中提取1-3个适合作为配图的中文场景描述..."
- 返回: 中文图片描述文本

**第 2 步: 用户编辑提示词**
- 前端 Workspace.vue 展示可编辑的提示词文本框
- 用户可修改、补充或完全重写图片提示词
- 支持多条提示词, 每条生成一张图片

**第 3 步: 提交 DashScope 任务**
- 调用 `POST /image/generate` 提交图片生成请求
- 后端调用 DashScope wanx-v1 文生图异步 API
- 返回 taskId 用于后续状态查询
- wanx-v1 参数: model, prompt, size, n (图片数量)

**第 4 步: 轮询任务状态**
- 前端定时轮询 `GET /image/generate/status/{taskId}`
- 显示进度条指示生成进度
- 状态流转: PENDING -> RUNNING -> SUCCEEDED / FAILED
- 轮询间隔: 2-3 秒

**第 5 步: 下载图片到 COS**
- 任务完成后, 调用 `POST /image/download`
- 从 DashScope 返回的 URL 下载图片
- 上传到 腾讯云 COS 对象存储, 生成唯一文件路径
- 返回 COS 访问 URL

**第 6 步: 创建素材资产**
- 在 `media_asset` 表创建记录
- 记录: 文件名, COS Key, 文件大小, MIME类型, 来源(AI_GENERATED)
- 自动标签: 从内容中提取主题 + 平台 + 场景关键词

**第 7 步: 关联到内容**
- 在 `content_media_rel` 关联表中创建记录
- 绑定 content_id 和 media_asset_id
- 前端图片网格中可选择/取消选择图片

- 抖音: 强调短视频脚本结构、口播节奏
- 公众号: 强调深度内容、排版结构
- 朋友圈: 强调简短有力、生活化

## 8. A/B 评测中心

### 8.1 功能概述

A/B 评测中心允许用户对比不同 AI 生成的内容质量。通过创建评测任务, 系统使用 LLM 作为裁判从六个维度评分, 自动消除位置偏差, 生成详尽的评测报告。

### 8.2 REST 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/eval/task` | 创建评测任务 |
| POST | `/eval/task/{id}/run` | 运行评测任务 (异步) |
| GET | `/eval/task/{id}/report` | 获取评测报告 |
| GET | `/eval/tasks` | 获取评测任务列表 |

### 8.3 评测流程

```text
┌─────────────────────────────────────────────────────────────────┐

│                                                                 │

│     POST /eval/task                                             │

│  2. 运行评测 (异步)                                              │
│     POST /eval/task/{id}/run                                    │
│     ┌─────────────────────────────────────────────────────┐     │
│     │  EvalServiceImpl.runTask()                          │     │
│     │                                                     │     │
│     │  a. 使用 configA 生成内容 outputA                    │     │
│     │  b. 使用 configB 生成内容 outputB                    │     │

│  3. LLM 裁判评判                                                 │

│     │  │ 维度         │ 权重   │                           │     │
│     │  ├─────────────┼────────┤                           │     │
│     │  │ relevance   │ 0.25   │  相关性                   │     │
│     │  │ style       │ 0.25   │  风格适配                 │     │
│     │  │ fluency     │ 0.15   │  流畅度                   │     │
│     │  │ creativity  │ 0.15   │  创意性                   │     │
│     │  │ format      │ 0.10   │  格式规范                 │     │
│     │  │ faithfulness│ 0.10   │  忠实度                   │     │
│     │  └─────────────┴────────┘                           │     │
│     └─────────────────────────────────────────────────────┘     │
│                                                                 │
│  4. 生成评测报告                                                 │

```

### 8.4 位置偏差消除

LLM 裁判在评判时可能存在位置偏差 (Position Bias), 即倾向于给出现在前面的内容更高分数。系统通过以下方式消除:

```text
第一轮评判:
  输入: "请比较以下两篇内容 A 和 B..."
  A = outputA (原始位置)
  B = outputB (原始位置)
  结果: scoreA1, scoreB1

第二轮评判 (随机交换):
  输入: "请比较以下两篇内容 A 和 B..."
  A = outputB (交换位置)
  B = outputA (交换位置)
  结果: scoreA2(实际为B), scoreB2(实际为A)

最终结果:
  finalScoreA = (scoreA1 + scoreB2) / 2
  finalScoreB = (scoreB1 + scoreA2) / 2
```

### 8.5 评测报告内容

- **胜负统计**: winsA / winsB / ties (A赢/B赢/平局次数)
- **维度平均分**: 六个维度各自的 A/B 平均分数
- **最佳案例**: 得分最高的生成内容及其评分详情
- **最差案例**: 得分最低的生成内容及其评分详情
- **评分分布**: 各维度的分数分布情况

## 9. 内容安全

### 9.1 功能概述

内容安全模块贯穿整个内容生成流程, 在输入阶段和输出阶段分别进行安全检查。采用责任链模式 (Chain of Responsibility), 支持多种检查策略的灵活组合和扩展。

### 9.2 责任链架构

```text
┌─────────────────────────────────────────────────────────────────┐
│                  GuardrailService 责任链                         │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                    输入检查链 (checkInput)                  │  │
│  │                                                           │  │
│  │  ┌──────────────┐    ┌──────────────┐    ┌─────────────┐ │  │
│  │  │ 敏感词过滤器  │───>│ 输入长度校验  │───>│ 后续可扩展  │ │  │
│  │  │              │    │              │    │  检查器      │ │  │
│  │  └──────────────┘    └──────────────┘    └─────────────┘ │  │
│  │                                                           │  │
│  │  任一环节检查不通过 -> 立即返回错误, 中断后续流程           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                    输出检查链 (checkOutput)                 │  │
│  │                                                           │  │
│  │  ┌──────────────┐    ┌──────────────┐    ┌─────────────┐ │  │
│  │  │ 敏感词过滤器  │───>│ LLM输出质量  │───>│ 后续可扩展  │ │  │
│  │  │              │    │  检查        │    │  检查器      │ │  │
│  │  └──────────────┘    └──────────────┘    └─────────────┘ │  │
│  │                                                           │  │
│  │  输出检查不通过 -> 记录日志 + 标记内容为需审核             │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 9.3 检查策略详解

**敏感词词典过滤:**
- 预加载敏感词词典到内存
- 使用高效匹配算法遍历文本
- 支持精确匹配和模糊匹配
- 命中敏感词时返回: 命中词汇、位置、违规等级

**输入长度校验:**
- 验证用户输入的主题、关键词等字段不超过最大限制
- 防止超长输入导致的资源消耗
- 各字段有独立的长度限制配置

**LLM 输出质量检查:**
- 将 AI 生成的内容发送给 LLM 进行质量评估
- 检查项: 是否含有不当内容、是否偏离主题、是否含有幻觉信息
- 评估结果: 通过 / 需人工审核 / 直接拦截

### 9.4 日志记录

| trigger_word | 触发词 (如有) |
| user_id | 用户 ID |
| content_id | 关联内容 ID |
| detail | 详细信息 |
| create_time | 检查时间 |

## 11. LLM Provider 架构

### 11.1 功能概述

LLM Provider 架构采用策略模式, 通过 LlmRouter 统一管理多个 LLM 提供商。支持 DeepSeek、Qwen、GLM 三个主要提供商, 可灵活切换和扩展。每个 Provider 封装了各自的 API 调用逻辑、模型配置和特殊处理。

### 11.2 架构设计

```text
┌─────────────────────────────────────────────────────────────────┐
│                     LLM Provider 架构                            │
│                                                                 │
│  LlmRouter                                                      │
│  ├── HashMap<LlmProvider, LlmProviderService>                   │
│  │                                                              │
│  │   ┌───────────────────────────────────────────────────┐      │
│  │   │  LlmProviderService (接口)                        │      │
│  │   │                                                   │      │
│  │   │  + chat(prompt, config): String                   │      │
│  │   │  + chatStream(prompt, config): Flux<String>       │      │
│  │   │  + getProvider(): LlmProvider                     │      │
│  │   └───────────────────────────────────────────────────┘      │
│  │            ▲              ▲              ▲                   │
│  │            │              │              │                   │
│  │   ┌────────────┐  ┌────────────┐  ┌────────────┐           │
│  │   │ DeepSeek   │  │   Qwen     │  │   GLM      │           │
│  │   │ LlmProvider│  │ LlmProvider│  │ LlmProvider│           │
│  │   └────────────┘  └────────────┘  └────────────┘           │
│  │                                                              │
│  │  chat(prompt, config) {                                      │
│  │    LlmProviderService svc = providers.get(config.provider);  │
│  │    return svc.chat(prompt, config);                          │
│  │  }                                                           │
│  └──────────────────────────────────────────────────────────────│
└─────────────────────────────────────────────────────────────────┘
```

### 11.3 各 Provider 详解

**DeepSeekLlmProvider:**
- 支持模型: `deepseek-chat`, `deepseek-reasoner`
- API 格式: OpenAI 兼容格式
- 特殊处理: 过滤 `reasoning_content` 字段和 `<think>` 标签
  - deepseek-reasoner 模型返回的响应中包含推理过程 (reasoning_content)
  - 在流式输出时需过滤这些内容, 只向前端推送最终回答
  - 正则过滤: 移除 `<think>...</think>` 标签包裹的内容
- 默认 baseUrl: `https://api.deepseek.com`

**QwenLlmProvider:**
- 支持模型: `qwen-plus`, `qwen-max`
- API 格式: DashScope 兼容 API (OpenAI 格式)
- baseUrl: `https://dashscope.aliyuncs.com/compatible-mode`

## 12. 向量存储与嵌入

### 12.1 功能概述

向量存储与嵌入模块提供文本向量化和向量检索能力, 是 RAG 知识库和内容相似度搜索的基础设施。使用 Qwen text-embedding-v3 模型生成 1024 维嵌入向量, 使用 PostgreSQL pgvector 扩展进行高效的向量存储和相似度检索。

### 12.2 嵌入服务

```text
┌───────────────────────────────────────────────────────────────┐
│              EmbeddingServiceImpl.java                         │
│                                                               │
│  模型: Qwen text-embedding-v3                                 │
│  维度: 1024                                                   │
│  API: DashScope 兼容 API                                      │
│  API Key: 独立配置 (与 LLM 调用分开)                          │
│                                                               │
│  embed(text: String): float[1024]                             │
│  embedBatch(texts: List<String>): List<float[1024]>           │
│                                                               │
│  调用流程:                                                     │
│  1. 构建请求: model="text-embedding-v3", input=text            │
│  2. POST https://dashscope.aliyuncs.com/compatible-mode/...   │
│  3. 解析响应中的 embedding 数组                                 │
│  4. 返回 1024 维浮点数组                                       │
└───────────────────────────────────────────────────────────────┘
```

### 12.3 向量存储服务

```text
┌───────────────────────────────────────────────────────────────┐
│            PgVectorStoreServiceImpl.java                       │
│                                                               │

│                                                               │
│  向量集合 (Collections):                                       │
│  ┌─────────────────────┬─────────────────────────────┐        │
│  │ sf_kb_chunks        │ 知识库文档分块向量            │        │
│  │                     │ - chunk_id                   │        │
│  │                     │ - vector (1024维)            │        │
│  │                     │ - metadata (kb_id, doc_id)   │        │
│  ├─────────────────────┼─────────────────────────────┤        │
│  │ sf_content_vectors  │ 生成内容向量                  │        │
│  │                     │ - content_id                 │        │

│  核心操作:                                                     │
│  ┌────────────────────────────────────────────────────┐       │
```

### 12.4 余弦相似度检索

```text
pgvector 余弦相似度检索示例:

SELECT chunk_id, metadata, 
       1 - (vector <=> $1) AS similarity
FROM sf_kb_chunks
WHERE metadata->>'kb_id' = $2
ORDER BY vector <=> $1
LIMIT $3;

参数:
  $1: 查询向量 (1024维)
  $2: 知识库 ID (过滤条件)
  $3: Top-K 数量

<=> 操作符: pgvector 提供的余弦距离运算符
similarity = 1 - cosine_distance (值越大越相似)
```

## 13. 数据库设计

### 13.1 概述

SocialFlow 采用双数据库架构:
- **MySQL**: 存储所有关系型业务数据
- **PostgreSQL**: 通过 pgvector 扩展存储向量数据

ID 生成策略: 使用 MyBatis-Plus 的 Snowflake 雪花算法生成分布式唯一 ID。由于 JavaScript Number 类型精度限制 (最大安全整数 2^53-1), Long 类型 ID 在 JSON 序列化时自动转为 String 类型。

全局功能:
- **逻辑删除**: 所有业务表包含 `is_deleted` 字段, 删除操作为逻辑删除
- **自动填充**: MyBatis-Plus MetaObjectHandler 自动填充 `create_time`, `update_time`

### 14.3 SSE 组合式函数 (useSse.ts)

```text
┌───────────────────────────────────────────────────────────────┐
│                    useSse.ts 实现原理                          │
│                                                               │
│  为什么不用 EventSource?                                       │
│  - EventSource 不支持自定义请求头                              │
│  - 无法携带 Authorization: Bearer <token>                     │
│  - 不支持 POST 方法                                           │
│                                                               │
│  使用 fetch API 实现:                                          │
│  ┌─────────────────────────────────────────────────────┐      │
│  │  const response = await fetch(url, {                │      │
│  │    method: 'POST',                                  │      │
│  │    headers: {                                       │      │
│  │      'Authorization': `Bearer ${token}`,            │      │
│  │      'Content-Type': 'application/json',            │      │
│  │      'Accept': 'text/event-stream'                  │      │
│  │    },                                               │      │
│  │    body: JSON.stringify(params)                      │      │
│  │  });                                                │      │
│  │                                                     │      │
│  │  const reader = response.body.getReader();          │      │
│  │  const decoder = new TextDecoder();                 │      │
│  │                                                     │      │
│  │  while (true) {                                     │      │
│  │    const { done, value } = await reader.read();     │      │
│  │    if (done) break;                                 │      │
│  │    const text = decoder.decode(value);              │      │
│  │    // 解析 SSE 格式: "data: {...}\n\n"              │      │
│  │    parseSSE(text);                                  │      │
│  │  }                                                  │      │
│  └─────────────────────────────────────────────────────┘      │
│                                                               │
│  返回值 (Composition API):                                     │
│  {                                                            │
│    content: Ref<string>,    // 累积的完整文本                  │
│    isLoading: Ref<boolean>, // 是否正在接收                    │
│    error: Ref<string>,      // 错误信息                       │
│    stage: Ref<string>,      // 当前阶段 (多Agent)             │

│    stop: () => void         // 中断连接                       │
│  }                                                            │
└───────────────────────────────────────────────────────────────┘
```

```yaml
    base-url: https://dashscope.aliyuncs.com/compatible-mode

  # ====== RAG 配置 ======
  rag:
    default-top-k: 10                    # 默认检索数量
    hyde-enabled: true                   # 是否启用 HyDE 查询改写
    rerank-enabled: false                # 是否启用 Rerank 重排序
    rrf-k: 60                            # RRF 融合参数 k
    max-context-length: 4000            # 最大上下文长度(字符)

  # ====== 图片生成配置 ======
  image:
    provider: dashscope                  # 图片生成提供商
    model: wanx-v1                      # 文生图模型
    dashscope-api-key: ${DASHSCOPE_API_KEY}  # DashScope API Key
    default-size: "1024*1024"           # 默认图片尺寸
    default-n: 4                         # 默认生成数量
    poll-interval-ms: 3000              # 状态轮询间隔(毫秒)
    max-poll-attempts: 60               # 最大轮询次数

  # ====== 发布配置 ======
  publish:
    wechat-mp:
      app-id: ${WECHAT_MP_APP_ID}      # 微信公众号 AppID
      app-secret: ${WECHAT_MP_APP_SECRET}  # 微信公众号 AppSecret
      auto-publish-enabled: true        # 是否启用自动发布

# ====== Spring 框架配置 ======
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/socialflow
    username: ${MYSQL_USERNAME}
    password: ${MYSQL_PASSWORD}
    driver-class-name: com.mysql.cj.jdbc.Driver

  redis:
    host: localhost
    port: 6379
    password: ${REDIS_PASSWORD}

# ====== Sa-Token 配置 ======
sa-token:
  token-name: Authorization
  token-prefix: Bearer
  timeout: 86400                         # Token 有效期(秒) 24小时
  is-concurrent: true                    # 允许同账号并发登录
```
