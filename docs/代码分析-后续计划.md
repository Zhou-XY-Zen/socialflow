## 后续计划 C · LLM 集成 321 条规约进审查 Prompt（待做）

### 背景

目前 `CodeReviewPrompts` 里没有引入阿里黄山版规约正文，LLM 审查依赖模型自己对 Java 规约的了解，**审查结果里的 `rule_ref` 字段是空的**。
若接入 321 条规约，LLM 可以在 Finding 里明确引用"违反了第 X.Y.Z 条"，审查权威性显著提升。

### 为什么先不做

1. **Prompt 长度瓶颈**：321 条完整正文 ≈ 80K tokens，一次塞不进 DeepSeek 的 32K 上下文
2. **路由逻辑成本高**：必须先按文件类型 / category 做"规约子集挑选"，挑错一条就会污染审查结论
3. **影响已跑通的 Map-Reduce 链路**：现在审查流程刚稳定，不能在同一波改 Prompt

### 落地路径（想清楚了，待触发）

前置步骤（**选项 B 的工作**）：

1. 建 `rule_library` 表
   - 字段：`id / code / top_category / sub_category / level / title / body / description / example_good / example_bad / full_text / create_time / update_time`
   - 索引：`uk_code`（code 唯一）、`idx_top_sub`（按大类小节查）、`idx_level`
2. V13 Flyway migration：建表，不灌数据
3. Entity / Mapper / Service / Controller
4. `RuleLibraryInitRunner`（ApplicationRunner）启动时：
   - 若表为空 → 从 classpath 读 `rules/huangshan_rules.json` 全量导入
   - 若非空 → 按 code 做 upsert（支持后续版本升级）
5. 暴露 `GET /code-analysis/rules`（分页 + 按 category / level / keyword 查询）
6. 前端 Rules.vue 改为从 API 加载（rules-data.ts 保留到 AI 审查投入生产后再移除）

C 的主步骤（**Prompt 集成**）：

7. 在 `CodeReviewPrompts` 新增一个注入点：
   - 入参：`category`（如 "MySQL-索引"）或 `fileType`（如 .sql / .java / .xml）
   - 内部按映射挑选该类目下相关规约（单次挑 20-40 条，控制在 6K tokens 以内）
   - 把 "code + title + body + example_good + example_bad" 拼进 System Prompt 的"规约参考"区
8. 审查 Prompt 要求 LLM 在 Finding 里输出 `rule_ref`（格式 `阿里手册黄山版 X.Y.Z`）
9. 文件类型 → 规约 category 的映射表（硬编码）：
   - `.java` → 编程规约·命名 / OOP / 集合 / 并发 / 控制 / 异常 / 日志
   - `.sql / *.xml (mybatis)` → MySQL·建表 / 索引 / SQL
   - `Test.java / *Tests.java` → 单元测试
   - 通用安全规约 → 所有文件都注入一部分（跨站、SQL 注入、加密等）
10. 上线后监控 `llm_call_log.prompt_tokens`，评估 Prompt 扩张后的 token 消耗变化

### 触发条件

以下任一条件达成，启动 C：

- 审查结论里用户反馈"希望看到违反了哪条具体规约"
- 要让 Rules.vue 页面支持"用户自定义 / 启停某条"
- 黄山版升级到 1.8+ 或嵩山版，需要热更新规约库

### 风险提示

- 规约子集挑选算法一定要做单元测试（不同文件类型 × 不同 category）
- Prompt token 扩张后，成本核算要重新算（目前 token 记录已就位，可观测）
- DeepSeek 返回的 `rule_ref` 可能幻觉不存在的编号 → Finding 入库时校验 code 是否在 rule_library 里存在，不存在则清空

## 其他想法池（更远的）

- Finding 反馈闭环：用户在详情页标记 "无效 Finding" → 回写统计指标，用于长期优化 Prompt
