## 1. 项目总览与技术选型

### 1.1 项目是做什么的

SocialFlow 是一个 **AI 社媒内容运营平台**。用户输入一个主题（如"春季护肤推荐"），系统自动为小红书、抖音、公众号等不同平台生成风格各异的文案。

### 1.2 技术栈一览

| 层次 | 技术 | 为什么选它 |
|------|------|-----------|
| 后端框架 | Spring Boot 3.3 + JDK 21 | 最新 LTS，企业标配 |
| AI 编排 | LangChain4j 1.0 | Java 生态最成熟的 LLM 编排框架 |
| ORM | MyBatis-Plus 3.5 | 比原生 MyBatis 少写 90% SQL |
| 认证 | Sa-Token 1.39 | 轻量级 JWT，比 Spring Security 简单 |
| 前端 | Vue 3 + TypeScript + Element Plus | 响应式 + 类型安全 + 丰富组件库 |
| 数据库 | MySQL 8.0 | 关系型数据，成熟稳定 |
| 向量库 | pgvector / Milvus | 存储文本向量，支持语义搜索 |
| 缓存 | Redis 7 | 会话管理、限流、缓存 |
| 对象存储 | 腾讯云 COS | 存储图片、视频、文档 |

### 1.3 整体架构图

```
浏览器（Vue3 前端）
    │
    │  HTTP / SSE
    ▼
Spring Boot 后端
    ├── web 层：接收请求，校验参数
    ├── service 层：业务编排
    │     ├── 业务服务：内容管理、知识库、发布、AI配图...
    │     └── AI 服务：LLM 调用、RAG、护栏、Agent...
    ├── dao 层：数据库操作
    └── model 层：数据结构定义
         │
    ┌────┴────┐
  MySQL    Redis    COS      pgvector    LLM APIs      DashScope
 (数据)   (缓存)   (文件)   (向量)    (DeepSeek等)   (AI配图)
```

## 2. Maven 多模块架构

### 2.1 为什么要用多模块

如果把所有代码放在一个项目里，会出现以下问题：
- **耦合严重**：Controller 直接 import Mapper 类，绕过了 Service 层
- **职责混乱**：一个包里既有 Entity 又有 Controller
- **编译慢**：改一行代码需要重新编译整个项目

Maven 多模块把项目按职责拆分，每个模块只做一件事：

```
socialflow/                    ← 父工程（只管理依赖版本）
├── socialflow-common/         ← 公共工具（不依赖任何业务）
├── socialflow-model/          ← 数据结构（Entity/DTO/VO）
├── socialflow-dao/            ← 数据库操作（Mapper）
├── socialflow-service/        ← 业务逻辑（AI + 业务）
├── socialflow-web/            ← 接口入口（Controller）
└── socialflow-admin/          ← 启动模块（main 方法）
```

### 2.2 模块依赖链

```
admin → web → service → dao → model → common
```

这条链是**单向的**，不能反向依赖。例如：
- common **不能** import model 中的类（因为 common 不依赖 model）
- dao **不能** import service 中的类（因为 dao 不依赖 service）

这种设计叫做**依赖倒置**，好处是：
- 改动 Controller 不会影响 Service
- 改动 Service 不会影响 Dao
- 每一层只关心自己和下一层

### 2.3 父 POM 的作用

父 POM (`pom.xml`) 不包含任何 Java 代码，它只做两件事：

**第一：声明模块列表**
```xml
<modules>
    <module>socialflow-common</module>
    <module>socialflow-model</module>
    <module>socialflow-dao</module>
    <module>socialflow-service</module>
    <module>socialflow-web</module>
    <module>socialflow-admin</module>
</modules>
```

**第二：统一管理依赖版本（dependencyManagement）**
```xml
<properties>
    <java.version>21</java.version>
    <langchain4j.version>1.0.0-beta3</langchain4j.version>
    <mybatis-plus.version>3.5.9</mybatis-plus.version>
    <sa-token.version>1.39.0</sa-token.version>
    ...
</properties>
```

为什么版本要在父 POM 里统一管理？因为如果每个子模块自己写版本号，很容易出现同一个依赖在不同模块用了不同版本，导致冲突。统一管理后，子模块只需要写 groupId 和 artifactId，不需要写 version。

## 3. 公共模块 common

common 模块是整个项目的"地基"，它定义了所有模块共用的基础设施。

### 3.1 统一返回体 R<T>

**问题**：如果每个接口返回格式不同，前端怎么判断请求是成功还是失败？

**解决方案**：定义一个统一的响应格式：

```java
public class R<T> implements Serializable {
    private Integer code;       // 状态码：200=成功，其他=失败
    private String message;     // 提示信息
    private T data;             // 业务数据（泛型，可以是任何类型）
    private Long timestamp;     // 响应时间戳
}
```

**前端收到的 JSON 永远长这样**：
```json
{
  "code": 200,
  "message": "ok",
  "data": { "id": 1, "title": "...", "body": "..." },
  "timestamp": 1712000000000
}
```

**使用方式**（在 Controller 中）：
```java
// 成功，返回数据
return R.ok(contentVO);

// 成功，无数据
return R.ok();

// 失败，自定义消息
return R.fail("文案生成失败");

// 失败，使用预定义错误码
return R.fail(ResultCode.QUOTA_EXCEEDED);
```

**为什么用泛型 `<T>`？**

`R<ContentVO>` 表示 data 字段是 ContentVO 类型，`R<List<String>>` 表示 data 是字符串列表。泛型让一个类适配所有返回类型，不需要为每种数据写一个返回类。

**`@JsonInclude(NON_NULL)` 的作用**：如果 data 为 null（比如删除操作没有返回数据），JSON 中就不会出现 `"data": null`，减少响应体积。

### 3.2 业务状态码 ResultCode

```java
public enum ResultCode {
    // -------- HTTP 标准码 --------
    SUCCESS(200, "ok"),
    BAD_REQUEST(400, "invalid request"),
    UNAUTHORIZED(401, "not logged in"),
    NOT_FOUND(404, "resource not found"),
    SERVER_ERROR(500, "internal server error"),

    // -------- AI 业务码（2xxx） --------
    AI_CALL_FAILED(2001, "AI call failed"),
    GUARDRAIL_BLOCKED(2002, "guardrail blocked the request"),
    QUOTA_EXCEEDED(2003, "quota exceeded"),

    // -------- 知识库码（3xxx） --------
    KB_NOT_FOUND(3001, "knowledge base not found"),
    EMBEDDING_FAILED(3004, "embedding failed"),

    // -------- 内容码（4xxx） --------
    CONTENT_NOT_FOUND(4001, "content not found"),
    PUBLISH_FAILED(4002, "publish failed");
}
```

**为什么要分段编号？**
- 1xxx = 通用业务错误
- 2xxx = AI 相关错误
- 3xxx = 知识库相关
- 4xxx = 内容相关

前端可以根据 code 的千位数判断错误属于哪个域。比如收到 2002，前端就知道是 AI 安全护栏拦截了请求，可以弹出对应的提示。

### 3.3 异常体系

```
RuntimeException
  └── BaseException               ← 所有业务异常的父类
        ├── BusinessException      ← 通用业务异常
        ├── ParamException         ← 参数校验失败（HTTP 400）
        ├── NotFoundException      ← 资源不存在（HTTP 404）
        ├── AiCallException        ← AI 调用失败（code 2001）
        ├── GuardrailException     ← 护栏拦截（code 2002）
        └── QuotaExceededException ← 额度超限（code 2003）
```

**为什么继承 RuntimeException 而不是 Exception？**

Java 有两种异常：
- `Exception`（受检异常）：必须用 try-catch 捕获，否则编译报错
- `RuntimeException`（非受检异常）：不强制捕获

业务异常用 RuntimeException 的好处是：Service 层抛出异常后，不需要在每个方法签名上写 `throws XxxException`，代码更干净。异常会自动向上冒泡，被 Web 层的全局异常处理器统一捕获。

**BaseException 的设计**：
```java
public abstract class BaseException extends RuntimeException {
    private final Integer code;   // 业务错误码

    public BaseException(ResultCode rc) {
        super(rc.getMessage());   // 消息存在 RuntimeException 的 message 字段
        this.code = rc.getCode(); // 错误码存在自己的 code 字段
    }
}
```

**使用方式**：在 Service 中随时抛出即可，不需要 try-catch：
```java
Content entity = contentMapper.selectById(id);
if (entity == null) {
    throw new NotFoundException("content not found: " + id);
}
```

### 3.4 枚举类设计

以 `PlatformType`（平台类型）为例：

```java
public enum PlatformType {
    XIAOHONGSHU("XIAOHONGSHU", "小红书", 20, 800, true),
    DOUYIN("DOUYIN", "抖音", 30, 300, true),
    WECHAT_MOMENT("WECHAT_MOMENT", "微信朋友圈", 0, 200, false),
    WECHAT_MP("WECHAT_MP", "微信公众号", 30, 3000, false);

    private final String code;
    private final String displayName;
    private final int titleMaxLength;
    private final int bodyMaxLength;
    private final boolean allowsEmoji;
}
```

**为什么把规则写在枚举里？**

每个平台有自己的内容约束（标题最长多少字、正文最长多少字、是否允许 Emoji）。把这些规则"挂"在枚举值上，后续代码可以直接：
```java
PlatformType platform = PlatformType.of("XIAOHONGSHU");
if (body.length() > platform.getBodyMaxLength()) {
    // 超出字数限制
}
```

不需要到处写 `if (platform == "小红书") { maxLength = 800; }`。新增平台时只需要在枚举里加一行，所有用到平台规则的地方自动生效。

### 3.5 加密工具 AesGcmUtil

用户在系统中配置自己的 AI API Key（如 DeepSeek 密钥），这些密钥必须加密存储：

```java
// 加密过程（简化）
public static String encrypt(String plaintext, String hexKey) {
    // 1. 生成 12 字节的随机初始化向量（IV）
    byte[] iv = new byte[12];
    new SecureRandom().nextBytes(iv);

    // 2. 用 AES-256-GCM 算法加密
    Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
    cipher.init(Cipher.ENCRYPT_MODE,
        new SecretKeySpec(key, "AES"),
        new GCMParameterSpec(128, iv));
    byte[] ciphertext = cipher.doFinal(plaintext.getBytes());

    // 3. 将 IV 和密文拼接后 Base64 编码
    byte[] combined = iv + ciphertext;  // 伪代码
    return Base64.encode(combined);
}
```

**为什么选 AES-256-GCM 而不是 AES-CBC？**
- GCM 模式自带完整性校验（AEAD），如果密文被篡改，解密时会自动失败
- CBC 模式需要额外加 HMAC 才能检测篡改
- GCM 是目前 TLS 1.3 和主流云服务商推荐的加密模式

### 3.6 Markdown 工具 MarkdownUtil

微信公众号的图文消息需要 HTML 格式，而 AI 生成的内容通常包含 Markdown 语法（如 `**加粗**`、`# 标题`）。MarkdownUtil 提供两种转换能力：

```java
public class MarkdownUtil {

    /**
     * 去除所有 Markdown 语法，返回纯文本。
     * 用于：微信文本群发（不支持任何格式标记）。
     */
    public static String stripMarkdown(String markdown) {
        // 移除标题标记 #、加粗 **、斜体 *、链接 []()、图片 ![]() 等
        // 返回干净的纯文本
    }

    /**
     * 将 Markdown 转换为 HTML。
     * 用于：微信公众号图文消息（需要 HTML 格式正文）。
     */
    public static String markdownToHtml(String markdown) {
        // 将 Markdown 语法转为对应的 HTML 标签
        // 例如 **加粗** → <strong>加粗</strong>
    }
}
```

**为什么需要两种方法？**

微信公众号有两种发布方式：
- **图文消息**（mpnews）：正文字段接受 HTML，所以用 `markdownToHtml()` 转换
- **文本群发**（text）：只接受纯文本，所以用 `stripMarkdown()` 去掉所有格式标记

这两个方法被 `WechatMpPublisher` 在发布流程中调用，确保内容格式符合微信 API 的要求。

## 4. 数据模型层 model

### 4.1 Entity / DTO / VO 三层模型

model 模块定义了三种 Java 对象，各有不同用途：

| 类型 | 全称 | 用途 | 命名 |
|------|------|------|------|
| Entity | 实体 | 与数据库表一一对应 | `Content`, `SysUser` |
| DTO | Data Transfer Object | 前端发给后端的请求体 | `ContentGenerateDTO` |
| VO | View Object | 后端返回给前端的响应体 | `ContentVO` |

**为什么需要三种对象？为什么不直接用 Entity？**

如果直接把 Entity 返回给前端，会有以下问题：
1. **安全问题**：`SysUser` Entity 包含 `passwordHash` 字段，直接返回就泄露了密码哈希
2. **字段不匹配**：Entity 的字段名可能和前端期望的不同（如 `aiModel` vs `model`）
3. **多余数据**：Entity 有 `isDeleted`、`updateTime` 等内部字段，前端不需要
4. **缺少字段**：前端可能需要计算字段（如 `guardrailWarnings`），Entity 里没有

所以：
- **Entity** = 数据库的忠实映射，字段和表完全对应
- **DTO** = 前端传什么我就接什么，带校验注解
- **VO** = 前端需要什么我就返什么，只暴露安全的字段

### 4.2 BaseEntity 公共字段

```java
public abstract class BaseEntity implements Serializable {

    @TableId(type = IdType.ASSIGN_ID)      // 主键策略：雪花算法
    private Long id;

    @TableField(fill = FieldFill.INSERT)   // 插入时自动填充
    private LocalDateTime createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)  // 插入和更新时自动填充
    private LocalDateTime updateTime;

    @TableLogic                             // 标记为逻辑删除字段
    @TableField(fill = FieldFill.INSERT)
    private Integer isDeleted;
}
```

每张表都有 id、create_time、update_time、is_deleted 四个公共字段。把它们抽到 BaseEntity 中，子类只需要 `extends BaseEntity`，不用重复写这四个字段。

**关键注解解释**：

| 注解 | 作用 |
|------|------|
| `@TableId(ASSIGN_ID)` | 主键用雪花算法生成（19 位 Long 型，全局唯一） |
| `@TableField(fill=INSERT)` | MyBatis-Plus 在 INSERT 时自动填充（由 MybatisMetaHandler 实现） |
| `@TableLogic` | 逻辑删除：调用 deleteById 时不会真删除，而是把 is_deleted 设为 1 |

**什么是逻辑删除？**

物理删除 = 执行 `DELETE FROM content WHERE id = 1`（数据永久消失）
逻辑删除 = 执行 `UPDATE content SET is_deleted = 1 WHERE id = 1`（数据还在，只是被标记）

逻辑删除的好处是数据可以恢复，且所有 SELECT 查询会自动加上 `WHERE is_deleted = 0` 条件。

### 4.3 DTO 的校验注解

```java
public class ContentGenerateDTO implements Serializable {

    @NotBlank               // 不能为空也不能只有空格
    @Schema(description = "主题", required = true)
    private String topic;

    @NotBlank
    @Schema(description = "XIAOHONGSHU / DOUYIN / WECHAT_MOMENT / WECHAT_MP")
    private String platform;

    @Schema(description = "期望字数")
    private Integer wordCount;

    @Schema(description = "LLM 温度，默认 0.7")
    private Double temperature;
}
```

当 Controller 方法参数上标了 `@Valid` 时，Spring 会自动检查 DTO 的校验注解：
```java
public R<ContentVO> generate(@Valid @RequestBody ContentGenerateDTO dto) { ... }
```
如果 `topic` 为空，Spring 自动返回 400 错误，不会进入方法体——无需手写 `if (dto.getTopic() == null)`。

## 5. 持久层 dao

### 5.1 MyBatis-Plus Mapper

传统 MyBatis 需要手写大量 XML SQL。MyBatis-Plus 通过继承 `BaseMapper<T>` 自动获得常用 CRUD 方法：

```java
@Mapper
public interface ContentMapper extends BaseMapper<Content> {
    // 不写任何方法，以下方法自动可用：
    // insert(entity)            插入
    // deleteById(id)            删除
    // updateById(entity)        更新
    // selectById(id)            按 ID 查
    // selectList(wrapper)       条件查询
    // selectPage(page, wrapper) 分页查询
}
```

**18 个 Mapper 只用了 18 个接口文件，零 SQL**——这就是 MyBatis-Plus 的威力。

### 5.2 自定义 SQL（当 BaseMapper 不够用时）

```java
@Mapper
public interface KnowledgeChunkMapper extends BaseMapper<KnowledgeChunk> {

    @Select("""
        SELECT * FROM knowledge_chunk
        WHERE kb_id = #{kbId}
          AND is_deleted = 0
          AND MATCH(content_text) AGAINST(#{query} IN NATURAL LANGUAGE MODE)
        ORDER BY MATCH(content_text) AGAINST(#{query} IN NATURAL LANGUAGE MODE) DESC
        LIMIT #{topK}
    """)
    List<KnowledgeChunk> fulltextSearch(@Param("kbId") Long kbId,
                                        @Param("query") String query,
                                        @Param("topK") int topK);
}
```

这条 SQL 用 MySQL 的全文索引（FULLTEXT + ngram 分词器）做关键词搜索，是 RAG 混合检索中"BM25 检索"那一路。相比纯 LIKE 模糊查询，全文索引利用了倒排索引和 BM25 评分算法，速度快且排序更精确。

### 5.3 自动填充配置

```java
@Component
public class MybatisMetaHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        LocalDateTime now = LocalDateTime.now();
        strictInsertFill(metaObject, "createTime", LocalDateTime.class, now);
        strictInsertFill(metaObject, "updateTime", LocalDateTime.class, now);
        strictInsertFill(metaObject, "isDeleted", Integer.class, 0);
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        strictUpdateFill(metaObject, "updateTime", LocalDateTime.class, LocalDateTime.now());
    }
}
```

每次 INSERT 时自动设置 createTime、updateTime、isDeleted。每次 UPDATE 时自动刷新 updateTime。开发者不需要手写 `entity.setCreateTime(LocalDateTime.now())`。

## 6. 服务层 service（业务）

### 6.1 内容服务 ContentServiceImpl

这是系统最核心的服务，generate 方法实现了 **5 步生成流水线**：

```
用户输入 → ① 安全检查 → ② RAG检索 → ③ 渲染Prompt → ④ 调用LLM → ⑤ 保存结果
```

```java
@Override
public ContentVO generate(Long userId, ContentGenerateDTO dto) {
    // 第 1 步：安全护栏检查用户输入
    guardrailService.checkInput(userId, dto.getTopic());

    // 第 2 步：如果指定了知识库，执行 RAG 检索
    String ragContext = null;
    if (dto.getKbId() != null) {
        ragContext = ragPipelineService.retrieveAsContext(
            userId, dto.getKbId(), dto.getTopic(), 5);
    }

    // 第 3 步：渲染 Prompt 模板
    Map<String, Object> vars = new HashMap<>();
    vars.put("topic", dto.getTopic());
    vars.put("keywords", dto.getKeywords());
    vars.put("productInfo", dto.getProductInfo());
    vars.put("wordCount", dto.getWordCount());
    promptService.render(dto.getTemplateId(), dto.getPlatform(), vars, ragContext);

    // 第 4 步：调用 LLM（TODO 待实现）
    // 第 5 步：检查输出 + 保存到数据库（TODO 待实现）
}
```

**为什么要分 5 步？**

每一步都是独立的关注点。如果把所有逻辑写在一个方法里，代码会变得冗长且难以维护。拆分后：
- 想换一个护栏引擎？只改 `guardrailService`
- 想换一个 RAG 策略？只改 `ragPipelineService`
- 想换一个 LLM 提供商？只改 `llmRouter`

**流式生成的健壮性增强**：

生成过程中可能遇到网络中断、LLM 服务超时等异常。系统做了三层防护：

1. **流式错误事件**：当 LLM 调用抛出异常时，不再静默失败，而是通过 `onErrorResume` 将错误信息包装为 SSE error 事件发送给前端，前端弹出错误提示而非无限等待。
2. **空内容检测**：如果 LLM 返回了空字符串（偶发现象），系统会发送警告事件提醒用户重试，避免用户以为生成成功实际内容为空。
3. **字数兜底强制**：Prompt 模板中加入了 `{{wordCount}}` 动态变量作为硬性字数限制。所有数据库中的模板都已更新，确保 LLM 生成的内容不会严重偏离用户期望的字数。

### 6.2 发布策略模式

系统需要向不同平台发布内容（微信、微博、小红书...），每个平台的发布方式不同。

**策略模式的三个角色**：

```
Publisher（接口）          ← 定义"发布"的统一规范
  ├── WeChatPublisher     ← 微信公众号：调用官方 API
  ├── WeiboPublisher      ← 微博：也调用 API
  └── XhsPublisher        ← 小红书：打包资源辅助发布

PublishRouter（路由器）    ← 根据平台类型选择对应的 Publisher
```

```java
@Component
public class PublishRouter {
    private final Map<PlatformType, Publisher> publishers = new HashMap<>();

    // Spring 自动注入所有 Publisher 实现
    public PublishRouter(List<Publisher> publisherBeans) {
        for (Publisher p : publisherBeans)
            publishers.put(p.platform(), p);
    }

    public Publisher get(PlatformType platform) {
        Publisher p = publishers.get(platform);
        if (p == null) throw new BusinessException("no publisher for " + platform);
        return p;
    }
}
```

**好处**：新增一个平台（如"B站"），只需要新建一个 `BilibiliPublisher implements Publisher`，加上 `@Component` 注解。不需要修改 PublishRouter 的任何代码——这就是"开闭原则"。

### 6.3 微信公众号发布（WechatMpPublisher）

微信公众号的发布是所有平台中最复杂的，因为微信 API 对**已认证**和**未认证**的公众号提供了不同的能力。

**3 级降级策略**：

```
尝试路径 ①（已认证 + 图文）
  └── 草稿箱 → freepublish 发表
       ↓ 失败
尝试路径 ②（已认证 + 简单图文）
  └── mpnews 群发
       ↓ 失败
尝试路径 ③（兜底）
  └── text 文本群发（纯文本，无格式）
```

**详细流程**：

1. **封面图自动生成**：如果内容没有配图，系统自动调用 AI 配图服务生成一张 900x383 的封面图（微信公众号图文消息的标准封面尺寸），上传到微信素材库获得 `media_id`
2. **内容格式转换**：AI 生成的 Markdown 内容通过 `MarkdownUtil.markdownToHtml()` 转为 HTML（图文消息需要），或通过 `MarkdownUtil.stripMarkdown()` 转为纯文本（文本群发需要）
3. **发布尝试**：按上述 3 级策略依次尝试，每级失败后自动降级

**未认证公众号的限制**：

未认证公众号没有"群发"和"发表"接口权限，因此：
- 前端发布中心会**禁用自动发布按钮**，提示用户需要先完成微信认证
- 后端直接返回提示信息，不会尝试调用受限 API

### 7.2 RAG 检索增强生成（8 步流水线）

RAG 是本项目最复杂的 AI 能力。当用户上传了行业文档，生成文案时可以参考这些文档。

**完整的 8 步流程**：

```
用户问题："写一篇小红书种草文案，推荐含氨基酸的洁面产品"
        │
        ▼
① HyDE 重写 ── 让 LLM 先生成一段"假设性回答"作为搜索词
        │
        ▼
② Embedding ── 把搜索词转成 1024 维向量
        │
        ▼
③ 向量搜索 ── 在 pgvector 中找语义最相似的 10 个文档片段
        │
        ▼
④ BM25 搜索 ── 在 MySQL FULLTEXT 中找关键词最匹配的 10 个片段
        │
        ▼
⑤ RRF 融合 ── 把两路结果合并去重，按综合得分排序
        │
        ▼
⑥ 补全数据 ── 从 MySQL 加载完整的片段文本
        │
        ▼
⑦ Rerank ── 用 Cross-Encoder 精排，从 20 个筛到 5 个
        │
        ▼
⑧ 注入 Prompt ── 把 5 个片段格式化为 [参考1] [参考2]... 放入提示词
```

**RRF 融合算法（代码实现）**：

```java
protected List<Long> fuseRrf(List<SearchHit> vectorHits, List<KnowledgeChunk> bm25Hits) {
    Map<Long, Double> scores = new LinkedHashMap<>();
    int k = 60;  // 平滑常数

    // 向量搜索结果
    int rank = 0;
    for (SearchHit hit : vectorHits) {
        rank++;
        long id = ((Number) hit.metadata().get("chunk_id")).longValue();
        scores.merge(id, 1.0 / (k + rank), Double::sum);
    }

    // BM25 搜索结果
    rank = 0;
    for (KnowledgeChunk chunk : bm25Hits) {
        rank++;
        scores.merge(chunk.getId(), 1.0 / (k + rank), Double::sum);
    }

    // 按得分降序排列
    return scores.entrySet().stream()
        .sorted(Map.Entry.<Long, Double>comparingByValue().reversed())
        .map(Map.Entry::getKey)
        .toList();
}
```

**RRF 的巧妙之处**：不同搜索引擎的分数范围不同（向量搜索可能是 0-1，BM25 可能是 0-100），直接比较无意义。RRF 只看排名位置，用 `1/(k + rank)` 计算分数。如果一个片段在两路搜索中都排名靠前，它的累加分数就最高。

### 7.3 AI 安全护栏（责任链模式）

护栏确保 AI 的输入合法、输出安全：

```
用户输入 → [输入长度检查] → [敏感词过滤] → [注入攻击检测] → LLM → [敏感词扫描] → [格式校验] → [内容安全] → 返回用户
```

**责任链模式**：

```java
public interface Guardrail {
    GuardrailRule rule();       // 规则名称
    Phase phase();              // INPUT 还是 OUTPUT
    int order();                // 执行顺序（小的先执行）
    Result evaluate(GuardrailContext ctx);  // 评估

    enum Action { PASS, WARNING, BLOCKED }  // 放行 / 警告 / 拦截
}
```

每个护栏是一个独立的 Spring Bean，服务启动时自动注册：

```java
@Service
public class GuardrailServiceImpl implements GuardrailService {
    private final List<Guardrail> guardrails;  // Spring 自动注入所有护栏 Bean

    private void runPhase(Phase phase, GuardrailContext ctx) {
        guardrails.stream()
            .filter(g -> g.phase() == phase)    // 筛选：只要当前阶段的规则
            .sorted(comparingInt(Guardrail::order))  // 排序：按优先级执行
            .forEach(g -> {
                Result result = g.evaluate(ctx);
                if (result.action() == BLOCKED) {
                    writeLog(ctx, g, result);        // 记录审计日志
                    throw new GuardrailException(...); // 拦截！抛异常终止
                }
                if (result.action() == WARNING) {
                    writeLog(ctx, g, result);        // 记录但不拦截
                }
                // PASS 则继续下一条规则
            });
    }
}
```

**好处**：
- 新增规则只需要写一个类 `implements Guardrail` + `@Component`
- 不需要修改 GuardrailServiceImpl 的任何代码
- 每条规则完全独立，可以单独测试、单独开关

### 7.4 Multi-Agent 多智能体协作

将文案生成拆分为 4 个专家角色：

```
Planner（策划）→ Writer（写作）→ Reviewer（审核）→ Optimizer（优化）
                                    ↑                    │
                                    └── 未通过时循环 ──────┘
```

每个 Agent 通过共享的 `AgentContext` 传递信息：

```java
public class AgentContext {
    private String topic;        // 用户输入的主题
    private String platform;     // 目标平台
    private String ragContext;   // RAG 参考资料
    private String plan;         // Planner 产出的策划方案
    private String draft;        // Writer 产出的初稿
    private String reviewResult; // Reviewer 产出的审核意见
    private int round;           // 当前第几轮迭代
}
```

**数据在 Agent 间的流转**：
1. Planner 读 topic + platform → 写入 plan
2. Writer 读 plan + ragContext → 写入 draft
3. Reviewer 读 draft → 写入 reviewResult（passed=true/false）
4. 如果 passed=false → Optimizer 读 draft + reviewResult → 更新 draft → 回到 Reviewer
5. 最多循环 3 轮，超过强制输出

### 7.5 AI 配图服务（ImageService）

AI 生成的文案往往需要配图。手动找图效率低且容易侵权，因此系统集成了阿里云 DashScope 的 **wanx-v1 文生图模型**，实现一键 AI 配图。

**核心类**：
- `ImageService` / `ImageServiceImpl`：配图业务逻辑
- `ImageController`：REST 接口，供前端调用

**完整流程**：

```
用户点击"AI配图"
    │
    ▼
① DeepSeek 提取配图 Prompt
    │  把用户的文案内容发给 LLM，要求提取出适合生成图片的英文描述词
    ▼
② DashScope 异步提交任务
    │  调用 wanx-v1 模型的 text-to-image API，提交绘图任务
    │  DashScope 返回一个 task_id（此时图片还没生成完）
    ▼
③ 轮询任务状态
    │  每隔几秒查询 task_id 的状态
    │  前端显示进度条，让用户知道正在生成中
    ▼
④ 下载图片到 COS
    │  任务完成后，从 DashScope 返回的 URL 下载图片
    │  上传到腾讯云 COS，生成永久访问地址
    ▼
⑤ 返回图片信息给前端
    └── 前端在"AI配图"面板中展示生成的图片
```

**为什么用异步轮询而不是同步等待？**

文生图模型生成一张图通常需要 10-30 秒，如果用同步请求，HTTP 连接会超时。DashScope 的设计是"提交任务 → 返回 task_id → 客户端轮询状态"，这是长时间任务的标准异步模式。前端通过进度条给用户即时反馈，避免用户以为系统卡死。

**为什么下载到 COS 而不是直接用 DashScope 的 URL？**

DashScope 返回的图片 URL 是临时链接，一段时间后会过期。下载到自己的 COS 后，图片永久可用，且后续发布到微信公众号等平台时可以直接引用。

### 8.2 全局异常处理器

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(NotLoginException.class)
    public ResponseEntity<R<Void>> handleNotLogin(NotLoginException e) {
        return ResponseEntity.status(401).body(R.fail(401, e.getMessage()));
    }

    @ExceptionHandler(GuardrailException.class)
    public ResponseEntity<R<Void>> handleGuardrail(GuardrailException e) {
        log.warn("护栏拦截: rule={} reason={}", e.getRuleName(), e.getMessage());
        return ResponseEntity.ok(R.fail(e.getCode(), e.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<R<Void>> handleUnknown(Exception e) {
        log.error("未处理异常", e);
        // 返回异常的类名 + 消息，方便前端和开发者快速定位问题
        String detail = e.getClass().getSimpleName() + ": " + e.getMessage();
        return ResponseEntity.status(500).body(R.fail(500, detail));
    }
}
```

**为什么 handleUnknown 返回 className + message 而不是固定的 "internal server error"？**

开发阶段遇到未预料的异常时，固定的错误信息无法帮助定位问题。返回 `异常类名: 异常消息`（如 `NullPointerException: userId is null`）可以让前端弹出有用的错误提示，开发者不用每次都去翻后端日志。生产环境可以通过配置开关切换为模糊提示。

**为什么 GuardrailException 返回 HTTP 200 而不是 400/500？**

因为请求本身是合法的（参数没问题、用户已登录），只是 AI 生成的内容触发了安全规则。这属于"业务层面的失败"而不是"协议层面的错误"。前端通过 `code: 2002` 判断是护栏拦截，弹出对应提示。

### 10.3 SSE 流式通信 useSse

这是前端最复杂的部分。为什么不用浏览器原生的 `EventSource`？因为 `EventSource` 只支持 GET 请求且无法设置自定义 Header（即无法传 Token）。

```typescript
export function useSse() {
  const streaming = ref(false)
  let abort: AbortController | null = null

  async function start(url: string, body: unknown, handlers: SseHandlers) {
    abort = new AbortController()
    streaming.value = true

    // 用 fetch 发 POST 请求，手动设置 Header
    const response = await fetch(url, {
      method: 'POST',
      signal: abort.signal,
      headers: {
        'Content-Type': 'application/json',
        Accept: 'text/event-stream',
        Authorization: `Bearer ${userStore.token}`,
      },
      body: JSON.stringify(body),
    })

    // 用 ReadableStream 逐块读取响应
    const reader = response.body!.getReader()
    const decoder = new TextDecoder('utf-8')
    let buffer = ''

    while (true) {
      const { done, value } = await reader.read()
      if (done) break

      buffer += decoder.decode(value, { stream: true })

      // SSE 格式：事件之间用 \n\n 分隔
      const chunks = buffer.split('\n\n')
      buffer = chunks.pop() || ''   // 最后一个可能不完整，留到下次

      for (const chunk of chunks) {
        parseEvent(chunk, handlers)  // 解析并分发到对应回调
      }
    }
  }
}
```

**在 Workspace.vue 中的使用**：

```typescript
const { streaming, start, stop } = useSse()

async function onGenerate() {
  body.value = ''  // 清空上次结果
  await start('/api/v1/content/generate-stream', form, {
    onMessage: (data) => { body.value += data.token },  // 逐字追加
    onStage: (data) => { stage.value = data.message },   // 显示当前阶段
    onDone: () => { stage.value = '生成完成' },
    onError: (data) => { stage.value = `错误：${data.message}` },
  })
}
```

用户看到的效果：点击"生成"按钮后，右侧文本区逐字出现 AI 生成的文案，就像 ChatGPT 的打字机效果。

### 11.2 设计决策

**为什么用雪花算法做主键而不是自增 ID？**
- 自增 ID 在分库分表时会冲突
- 自增 ID 暴露了业务量（ID 1000 说明只有 1000 条数据）
- 雪花 ID 是 19 位 Long 型，全局唯一，有序递增

**为什么 knowledge_chunk 有冗余的 kb_id 字段？**
```sql
CREATE TABLE knowledge_chunk (
    doc_id  BIGINT NOT NULL,      -- 属于哪个文档
    kb_id   BIGINT NOT NULL,      -- 属于哪个知识库（冗余！）
    ...
);
```
根据范式设计，`kb_id` 可以通过 `doc_id → knowledge_document.kb_id` 关联获取。但 RAG 检索时需要按 kb_id 过滤 chunk，如果每次都 JOIN document 表会很慢。冗余存储 kb_id 后，检索 SQL 只需要一张表，性能大幅提升。

**为什么 content 表的 JSON 字段用 String 而不是用子表？**
```java
private String generationParams;  // JSON: {topic, keywords, model, ...}
```
生成参数只需要保存一份快照，不需要按字段查询或修改。用 JSON 列可以灵活存储不同格式的参数，不需要为每种参数组合建表。

## 12. 配置体系

### 12.1 application.yml 分组

```yaml
socialflow:
  ai:
    default-provider: deepseek              # 默认 LLM 提供商
    guardrails:
      enabled: true                         # 护栏总开关
      max-input-tokens: 2000               # 输入最大 Token
    quota:
      daily-tokens-per-user: 100000         # 每人每日 Token 限额
      rate-limit-per-minute: 10             # 每分钟调用限制

  rag:
    chunk-size: 512                         # 分块大小（Token）
    chunk-overlap: 64                       # 分块重叠
    enable-hyde: true                       # 启用 HyDE 查询重写
    enable-rerank: true                     # 启用重排序

  vector:
    type: pgvector                          # 向量库类型
    host: localhost
    port: 5432

  image:
    api-key: ${DASHSCOPE_API_KEY:}          # DashScope API Key
    model: wanx-v1                          # 文生图模型
    poll-interval-ms: 3000                  # 轮询间隔（毫秒）
    poll-max-attempts: 40                   # 最大轮询次数
```

**`socialflow.image.*` 配置说明**：

这组配置专门用于 AI 配图功能（7.5 节），控制 DashScope 文生图服务的调用参数：
- `api-key`：通过环境变量 `DASHSCOPE_API_KEY` 注入，不硬编码在配置文件中
- `model`：使用阿里云通义万相的 wanx-v1 模型
- `poll-interval-ms` 和 `poll-max-attempts`：控制异步任务轮询的频率和上限，避免无限等待。默认 3 秒轮询一次，最多轮询 40 次（即最长等待约 2 分钟）

**为什么用 `${DB_PASSWORD:***REDACTED-DB-PASSWORD***}` 这种写法？**

`${变量名:默认值}` 是 Spring Boot 的占位符语法：
- 优先读取环境变量 `DB_PASSWORD`
- 如果环境变量不存在，使用默认值 `***REDACTED-DB-PASSWORD***`

敏感信息（密码、密钥）通过环境变量注入，不会硬编码在配置文件中，避免泄露到 Git 仓库。

## 13. 整体数据流

### 13.1 文案生成数据流

以"用户在工作台生成一篇小红书种草文案"为例，完整的数据流转：

```
[前端 Workspace.vue]
    │ 用户填写表单，点击"流式生成"
    │ useSse.start('/api/v1/content/generate-stream', form)
    ▼
[Axios 拦截器]
    │ 自动注入 Authorization: Bearer {token}
    ▼
[Nginx / Vite Proxy]
    │ /api 开头的请求转发到 localhost:8080
    ▼
[ContentController.generateStream]
    │ @Valid 校验参数
    │ StpUtil.getLoginIdAsLong() 获取用户 ID
    ▼
[ContentServiceImpl.generateStream]
    │
    ├── ① GuardrailService.checkInput(userId, topic)
    │       └── InputLengthGuardrail → SensitiveWordGuardrail → ...
    │
    ├── ② RagPipelineService.retrieveAsContext(userId, kbId, topic, 5)
    │       └── HyDE → Embedding → 向量搜索 → BM25 → RRF → Rerank
    │
    ├── ③ PromptService.render(templateId, platform, vars, ragContext)
    │       └── 加载模板 → 替换变量 → 注入 RAG 上下文 → 返回 messages[]
    │
    ├── ④ LlmRouter.get("deepseek").chatStream(messages, config)
    │       └── 调用 DeepSeek API，逐 token 返回 Flux<String>
    │
    └── ⑤ GuardrailService.checkOutput(userId, text, platform, ragContext)
            └── 敏感词扫描 → 格式校验 → 幻觉检测
    │
    ▼
[Spring WebFlux SSE]
    │ 每个 token 包装为 event: message, data: {"token": "一小段文字"}
    ▼
[前端 useSse.parseEvent]
    │ 解析 SSE 事件，调用 onMessage 回调
    ▼
[Workspace.vue onMessage]
    │ body.value += data.token  // 逐字追加到文本区
    ▼
[用户看到文案逐字出现]
```

## 总结

本项目的核心设计理念：

| 原则 | 在项目中的体现 |
|------|--------------|
| **统一规范** | R<T> 统一返回体、ResultCode 统一错误码、BaseEntity 统一公共字段 |
| **分层解耦** | 6 个 Maven 模块，单向依赖，改动互不影响 |
| **策略模式** | LlmRouter（LLM 切换）、PublishRouter（平台切换）——新增实现零修改 |
| **责任链** | Guardrail 护栏链——新增规则零修改 |
| **接口抽象** | VectorStoreService 屏蔽 pgvector/Milvus 差异——切换存储零修改 |
| **配置驱动** | 所有阈值（chunk-size、top-k、max-tokens、image poll 参数）都在 yml 中，改配置不改代码 |
| **安全优先** | AES-256-GCM 加密 API Key、环境变量注入密码、护栏体系保障 AI 安全 |
| **优雅降级** | 微信发布 3 级降级（draft → mpnews → text）、流式生成错误事件回传前端 |
| **异步模式** | DashScope 文生图任务提交 + 轮询，长任务不阻塞用户 |
